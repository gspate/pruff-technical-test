# Decisiones de Stack
Para partir el proyecto decidí utilizar Next.js con App Router. La razón principal es que facilita el manejo de rutas y me permite usar Server Actions, ahorrandome tiempo al no tener que armar endpoints intermedios solo para iniciar sesión o registrar usuarios. Para la base de datos elegi PostgreSQL y como ORM utilice Prisma 7, principalmente por su TypeSafety nativo y sintaxis más limpia frente a otros ORM (como Sequelize). Finalmente, para el envío de correos y reportes utilicé Resend, ya que su API es rápida, simple y evita tener que gestionar configuración de los servidores SMTP tradicionales (aparte ya tenía un dominio comprado, por lo que Resend me quedaba perfecto).

# Supuestos
Por temas de tiempo no implementé los filtros más específicos para las búsquedas de cada tipo de propiedad (por ejemplo, número de baños, precio, barrio, etc.), solo consideré la búsqueda por comuna. Además, la página que se scrapea en Portal Inmobiliario tiene el órden seteado por "Ordenar por: Más relevantes", por lo que las propiedades mostradas en el mi app siguen ese mismo órden.

# 1. Scraping
Implementé el scraping utilizando peticiones HTTP directas (fetch) combinadas con cheerio para parsear el HTML en lugar de usar navegadores pesados como Playwright. Tomé esta decisión porque levantar un navegador completo consume demasiada memoria y tarda varios segundos por petición, por lo que extraer el HTML mediante peticiones directas es casi instantaneo y permite que mi infraestructura en AWS sea muchisimo mas liviana y económica de escalar. Para evitar bloqueos tempranos, inyecto headers HTTP simulando ser un navegador humano estándar en Mac. Además, para lograr que nuestras búsquedas arrojaran exactamente los mismos resultados que la página original, usé las DevTools del navegador para extraer el formato exacto de los parámetros de URL que utiliza PortalInmobiliario internamente para realizar las búsquedas por comunas (ej.: "¿Por qué ese formato específico de búsqueda de comuna?" --> si escribo ´vitacura´ en el buscador de Portal Inmobiliario se rellena el campo de búsqueda con ´Vitacura, RM (Metropolitana)´, y estas búsquedas arrojan resultados distintos.). De esta forma aseguro que al escribir el mismo string en el buscador de Portal Inmobiliario que en mi app, tendré los mismos resultados.

# 2. Modelo de datos y esquema
- Usuarios: credenciales e informacion basica.
- Historial de búsquedas: registro detallado de los filtros, fecha y cantidad de resultados de cada busqueda, pensado netamente para analitica.
- Eventos: una tabla de auditoria transaccional para trazar acciones clave del usuario (como logins exitosos).
- DailyReportLog: un registro de seguridad clave para garantizar que los reportes diarios de cron no se envien dos veces.

Para los favoritos, solo guardo un puntero (la URL original y el ID externo) junto a una pequeña captura temporal en formato JSON (titulo, precio, foto). Usé JSON porque me da flexibilidad --> si mañana el portal cambia los datos que entrega, el campo lo soporta sin tener que modificar mis columnas fijas. Además evito adueñarme de información que no controlo.

# 3. Robustez y UX
Respecto a la robustez del scraper:
1. Si el portal cambia su HTML, el código está preparado para fallar de forma segura devolviendo una lista vacía, en lugar de botar el servidor.
2. Si el portal me bloquea o detecta trafico inusual (errores 403 o 429), capturo esos codigos y devuelvo un mensaje de error controlado a la interfaz indicando que se alcanzó un límite.
3. Si el portal simplemente no responde o se cae (error 500 o 504), implementé un algoritmo de reintentos exponenciales. El scraper espera un par de segundos y vuelve a intentar automáticamente, haciéndome tolerante a microcortes sin afectar al usuario.

Para la interfaz, decidí priorizar la consistencia visual y la percepción de rendimiento. En vez de usar el clásico texto de "Cargando..." que hace saltar todo el contenido de la pantalla, implementé Skeletons estrictos (tarjetas grises del tamaño exacto del resultado final) para evitar el molesto parpadeo visual (Cumulative Layout Shift) y dar la sensación de que la app responde mejor. Además, me enfoqué en mantener un diseño limpio y cohesivo en todas las vistas: definí una paleta de colores constante, una tipografía unificada y mantuve una coherencia estricta en los tamaños, sombras y radios de los botones e inputs. El objetivo detrás de esto era que la plataforma no solo funcionara bien por debajo, sino que al interactuar con ella se sintiera como un producto lo más terminado posible.

# 4. Trade-offs y que haría distinto
El mayor trade-off que asumí fue guardar una copia estática de la propiedad al marcarla como favorita. Sé que no es ideal porque me desincroniza de la realidad: si la casa baja de precio mañana o se arrienda, el usuario seguirá viendo la información vieja en su panel de favoritos.
Con más tiempo, mi tabla de favoritos solo guardaría la URL. Cuando el usuario entre a su panel, mi backend haría un re-scrapeo en vivo (en paralelo) de esas propiedades para mostrar siempre la información actualizada. Para que esto no sature la red ni sea lento, pondría una capa de Redis en medio para cachear esos resultados por un par de horas, garantizando datos reales sin duplicar estado permanentemente en mi disco. Y con más tiempo también me hubiese preocupado de ajustar la UI dependiendo del dispositivo con el que se estuviera visitando la página, ya que trabajé en todo el proyecto solo considerando la vista dekstop del browser, nunca la vista mobile.

Otro trade-off importante fue descartar el botón de ver en mapa (como el que tiene PortalInmobiliario). Al investigar, confirmé que no inyectan las coordenadas geográficas en la vista de lista. Obtenerlas me habría obligado a scrapear el detalle de cada propiedad por separado. Hacer 50 peticiones de golpe por cada búsqueda garantizaba un baneo casi inmediato por abuso de red, así que decidí priorizar la estabilidad de la plataforma y descartar esa feature.

# Infraestructura y Seguridad
Para proteger el buscador de abusos externos (dado que hace peticiones de scraping pesadas), implementé autenticación con JWT inyectados en cookies. El backend rechaza cualquier petición de búsqueda que venga sin una sesión válida.
Para el deploy decidí utilizar AWS ECS Fargate (Serverless) junto con Cloudflare Tunnel. Toda la aplicación está containerizada usando Docker, garantizando que el entorno de ejecución sea idéntico tanto en mi computador local como en producción. La imagen final empaqueta el servidor de Next.js y Prisma, se sube a AWS ECR, y Fargate la ejecuta. Tomé la decisión de usar Cloudflare Tunnel porque evito gestionar y pagar balanceadores de carga (ALB), configurar redes públicas complejas o administrar certificados SSL a mano. El backend corre en su contenedor de forma totalmente aislada e invisible al internet público, y Cloudflare redirige el tráfico filtrado mediante túnel reverso. La base de datos corre administrada en RDS PostgreSQL y los secretos se consumen dinámicamente desde AWS Secrets Manager. Finalmente, toda esta infraestructura (redes, bases de datos, contenedores y permisos) fue aprovisionada y gestionada 100% mediante Terraform (IaC), asegurando que el despliegue sea automatizado, inmutable y replicable. Para más detalles de la arquitectura + deploy ver el archivo con nombre ARQUITECTURA.md.
